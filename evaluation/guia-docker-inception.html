NIVEL 1

hello-world

¿Qué es una imagen? ¿Qué es un contenedor? Ciclo de vida mínimo: build → run → exit.

Conceptos

  • Imagen = plantilla de solo lectura
  • Contenedor = proceso aislado
  • Layer = cada instrucción RUN/COPY
  • FROM scratch = vacío absoluto

Comandos

  • docker build -t nombre .
  • docker run --rm nombre
  • docker images
  • docker history nombre
Dockerfile nivel1/ 
# FROM scratch = imagen vacía total (0 bytes base)
FROM busybox:latest

LABEL maintainer="tu_login@student.42.fr"

# CMD: proceso ejecutado al hacer docker run
# El contenedor sale (exit 0) cuando CMD termina
CMD ["echo", "Hola desde mi primer contenedor!"]
$ docker build -t mi-hello .
↓
FROM busybox → descarga layer base
LABEL → metadata (no crea layer)
CMD → comando por defecto
↓
$ docker run --rm mi-hello
↓
Hola desde mi primer contenedor! → exit 0 → contenedor destruido
NIVEL 2

Alpine Linux

Sistema base minimalista (~5MB). Aprende apk, ENV, RUN, layers y la diferencia entre imagen y contenedor.

Alpine vs Debian

  • Alpine: ~5MB, usa musl libc
  • Debian: ~70MB, compatible total
  • Alpine usa apk (no apt)
  • Ambas válidas en Inception 42

Nuevas instrucciones

  • ENV = variable en build Y runtime
  • ARG = solo en build
  • WORKDIR = mkdir + cd
  • COPY = copia host → imagen
Dockerfile nivel2-alpine/ 
FROM alpine:3.19

# ENV persiste en la imagen (visible con docker inspect)
ENV MI_VARIABLE="hola_42"

# RUN: se ejecuta en BUILD, crea un layer
# && encadena y limpiar caché reduce tamaño final
RUN apk update && \
    apk add --no-cache bash curl wget vim \
    && rm -rf /var/cache/apk/*

WORKDIR /app
COPY info.sh /app/info.sh
RUN chmod +x /app/info.sh

CMD ["/bin/bash", "/app/info.sh"]
💡
Cada instrucción RUN crea un layer nuevo. Encadenar comandos con && y limpiar la caché en el mismo RUN reduce el tamaño de la imagen final.
NIVEL 2

Debian Bullseye

La base que usarás en Inception. Aprende apt-get, ARG vs ENV y DEBIAN_FRONTEND=noninteractive.

Dockerfile nivel2-debian/ 
# Inception 42: usa debian:bullseye (penúltima estable)
FROM debian:bullseye

# ARG: solo en build (no persiste en imagen)
ARG VERSION="1.0"

# ENV: persiste en runtime
ENV APP_VERSION=$VERSION

# DEBIAN_FRONTEND=noninteractive: sin prompts en apt
RUN apt-get update && \
    DEBIAN_FRONTEND=noninteractive apt-get install -y \
        --no-install-recommends \
        bash curl wget \
    && apt-get clean \
    && rm -rf /var/lib/apt/lists/*

WORKDIR /app
COPY info.sh /app/info.sh
CMD ["/bin/bash", "/app/info.sh"]

ARG vs ENV

  • ARG: solo existe en build
  • ENV: persiste en imagen final
  • ARG para versiones, fechas
  • ENV para config de runtime

apt limpieza

  • apt-get clean → borra .deb
  • rm /var/lib/apt/lists/* → borra índices
  • --no-install-recommends → sin extras
  • Ahorra 30-50MB por imagen
NIVEL 2

Arch Linux

Rolling release, pacman, glibc, filosofía KISS. Solo educativo — Arch no está permitida en Inception 42, pero es perfecta para entender las diferencias entre distros.

⚠️
No válida para Inception 42. El subject exige debian:bullseye o alpine. Este nivel existe para comparar gestores de paquetes y entender por qué Debian es la elección correcta para el proyecto.

Diferencias clave

  • ~140MB base (vs 5MB Alpine)
  • Rolling release: sin versión fija
  • glibc: máxima compatibilidad
  • pacman: rápido, sin sub-comandos

musl vs glibc

  • Alpine usa musl: ligero, menos compat.
  • Arch y Debian usan glibc
  • php-fpm, mysql necesitan glibc
  • Por eso Inception usa Debian, no Alpine
Dockerfile nivel2-arch/ 
# archlinux:base = ~140MB, rolling release, glibc
FROM archlinux:base

ENV MI_VARIABLE="hola_42"
ENV OTRA_VAR="arch_btw"

# pacman -Syu: update BD + upgrade SIEMPRE juntos
# Nunca solo -Sy (rompe dependencias en rolling)
# --noconfirm = equivalente a -y en apt-get
RUN pacman -Syu --noconfirm && \
    pacman -S --noconfirm --needed \
        bash curl wget vim \
    && pacman -Scc --noconfirm
#          ^^^^
#          Limpia caché de pacman (paquetes + BD)

WORKDIR /app
COPY info.sh /app/info.sh
RUN  chmod +x /app/info.sh
CMD  ["/bin/bash", "/app/info.sh"]
Comparativa gestores de paquetes
Alpine apk add --no-cache curl

Debian apt-get install -y curl

Arch pacman -S --noconfirm curl


Limpiar caché:
Alpine rm -rf /var/cache/apk/*

Debian apt-get clean && rm /var/lib/apt/lists/*

Arch pacman -Scc --noconfirm
NIVEL 3

Python HTTP server

Primer proceso en foreground. Conceptos clave: EXPOSE, mapeo de puertos -p, y la diferencia crítica entre CMD y ENTRYPOINT.

Dockerfile nivel3-python/ 
FROM debian:bullseye

RUN apt-get update && \
    DEBIAN_FRONTEND=noninteractive apt-get install -y python3 \
    && apt-get clean && rm -rf /var/lib/apt/lists/*

WORKDIR /var/www/html
COPY index.html /var/www/html/index.html

# EXPOSE: documenta el puerto (NO lo abre)
EXPOSE 8000

# ENTRYPOINT: ejecutable fijo
# CMD: argumentos por defecto (sobreescribibles)
ENTRYPOINT ["python3", "-m", "http.server"]
CMD         ["8000"]
CMD vs ENTRYPOINT
docker run imagen → python3 -m http.server 8000
docker run imagen 9000 → python3 -m http.server 9000
Mapeo de puertos: -p HOST:CONTENEDOR
docker run -p 8080:8000 → localhost:8080 → contenedor:8000
⚡
Concepto clave para Inception: el proceso debe correr en foreground como PID 1. Si el proceso se daemoniza solo, el contenedor muere inmediatamente.
NIVEL 3

Nginx con conf propia

Nginx se daemoniza por defecto. Aprende a forzarlo en foreground con daemon off. Base directa del Nginx de Inception.

Dockerfile nivel3-nginx/ 
FROM debian:bullseye

RUN apt-get update && \
    DEBIAN_FRONTEND=noninteractive apt-get install -y nginx \
    && apt-get clean && rm -rf /var/lib/apt/lists/*

# Reemplaza la configuración por defecto
COPY nginx.conf /etc/nginx/nginx.conf
COPY html/ /var/www/html/

EXPOSE 80

# CRÍTICO: "daemon off" = nginx en foreground = PID 1
# Sin esto: nginx va al background → proceso muere → contenedor sale
CMD ["nginx", "-g", "daemon off;"]
Sin daemon off vs con daemon off
nginx → fork → worker (bg) → PID 1 termina → contenedor muere ✗

nginx -g "daemon off;" → PID 1 = nginx master → contenedor vive ✓
NIVEL 4

MySQL / MariaDB

Instalado desde paquetes (sin imagen oficial). Aprende VOLUME, HEALTHCHECK, inicialización con scripts y el patrón exec para PID 1.

Dockerfile nivel4-mysql/ 
# Para Inception: FROM debian:bullseye, NO mysql:latest
FROM debian:bullseye

RUN apt-get update && \
    DEBIAN_FRONTEND=noninteractive apt-get install -y \
        mariadb-server \
    && apt-get clean && rm -rf /var/lib/apt/lists/*

COPY entrypoint.sh /entrypoint.sh
RUN  chmod +x /entrypoint.sh

# VOLUME: los datos sobreviven al contenedor
VOLUME ["/var/lib/mysql"]
EXPOSE 3306

# HEALTHCHECK: docker sabe si el servicio está realmente listo
# Otros contenedores usan depends_on con condition: service_healthy
HEALTHCHECK --interval=10s --timeout=5s --retries=3 \
    CMD mysqladmin ping -h localhost

ENTRYPOINT ["/entrypoint.sh"]
entrypoint.sh (fragmento clave) bash 
# exec reemplaza el script por mysqld
# → mysqld es PID 1 (no el script bash)
# → recibe señales SIGTERM correctamente
exec mysqld --user=mysql --bind-address=0.0.0.0
💡
exec al final del entrypoint es el patrón correcto para Inception. Reemplaza el shell por el proceso final, que se convierte en PID 1 y puede recibir señales del sistema correctamente.
NIVEL 4

Redes y volúmenes

docker-compose con networks (bridge, DNS interno) y volumes named. Base de la arquitectura de Inception.

Networks bridge

  • Red virtual privada aislada
  • DNS interno: nombre servicio = hostname
  • web puede hacer curl http://db:3306
  • Prohibido --link en Inception

Volumes named

  • Datos persisten entre up/down
  • docker compose down NO los borra
  • down -v sí los borra
  • En Inception: bind mount al host
docker-compose.yml nivel4-red/ 
services:
  db:
    build: ../nivel4-mysql
    environment:
      MYSQL_ROOT_PASSWORD: rootpass
      MYSQL_DATABASE: mi_db
    volumes:
      - db_data:/var/lib/mysql   # volumen named
    networks:
      - red_interna
    healthcheck:
      test: ["CMD", "mysqladmin", "ping"]
      interval: 10s
      retries: 5

  web:
    build: ../nivel3-nginx
    ports: ["8080:80"]
    networks: [red_interna]
    depends_on:
      db:
        condition: service_healthy   # espera healthcheck ✓

volumes:
  db_data:
    name: nivel4_db_data

networks:
  red_interna:
    driver: bridge

Reglas de Inception 42

Cosas que el corrector va a revisar. Un error aquí = fallo inmediato.

🚫
Prohibido usar latest como tag de imagen. Siempre versión explícita: debian:bullseye, alpine:3.19.
🚫
Prohibido usar imágenes oficiales de servicio directamente: no mysql:8, no wordpress:latest, no nginx:alpine. Debes construir desde debian:bullseye o alpine.
🚫
Prohibido tail -f /dev/null o cualquier hack para mantener el contenedor vivo. El proceso real debe ser PID 1. Usa exec servicio --nodaemon.
  • ✅
    PID 1 correcto: usa exec en el entrypoint para que el proceso final sea PID 1. Recibe señales SIGTERM y SIGKILL correctamente.
  • ✅
    Credenciales en .env: todas las contraseñas y secrets en .env. El fichero .env en .gitignore. Proporciona .env.example en el repo.
  • ✅
    Solo TLSv1.2 y TLSv1.3: nginx.conf debe tener ssl_protocols TLSv1.2 TLSv1.3; explícitamente. No SSLv3, no TLSv1.0, no TLSv1.1.
  • ✅
    Solo puerto 443: Nginx solo expone HTTPS. No port 80 en el compose final de Inception.
  • ✅
    Red sin --link: usa networks: en docker-compose. Docker crea DNS interno automáticamente.
  • ✅
    Volúmenes bind mount al host: driver_opts: type: none, o: bind, device: /home/login/data/...
  • ✅
    restart: unless-stopped: los servicios deben reiniciarse automáticamente si el daemon de Docker arranca (simulado con reboot de la VM).
  • ✅
    Makefile con targets: all, build, up, down, clean, fclean, re mínimo.
